# Neptune Inspector H5 网络与协议审计

## 结论

当前实现没有自研 HTTP 客户端、重试器、代理层、URL 解析器或二进制协议解析器。
HTTP 请求仍统一交给浏览器原生 `fetch` 和 `URL` 处理；新增 WS 传输层只封装握手、heartbeat、重连和消息归类，不引入自研帧协议或二进制解析。

## 已落地的边界校验

- `logs`、`clients`、`metrics` 三类接口仍然通过单一的 `fetch` 边界访问。
- 响应体不再直接强转为 TypeScript 类型，而是在边界处使用 `zod` 做运行时校验。
- 校验失败会明确报出接口名和字段路径，避免脏数据静默进入 UI。
- WS 入站消息采用宽松 JSON 解码 + topic 分发，`command.send(ping)` 入口在前端先做 target 非空校验。
- `PUT /v2/clients:selected` 只做完整选择集提交，不在前端做增量合并或冲突解决。

## 保持现状的理由

- 该仓库只消费简单 JSON 快照，不涉及自定义帧、流式协议或压缩协议。
- 浏览器原生能力已经覆盖 HTTP、URL、JSON 这三项基础设施。
- `zod` 负责 HTTP 协议形状校验，WS 侧只做最小的消息路由和状态更新。

## 风险边界

- 上游接口字段变更会在前端立即暴露为可读错误。
- WS 端点消息结构如果继续演进，需要同步更新前端的消息归类和 inbox 文案，但不会影响 HTTP 快照路径。
